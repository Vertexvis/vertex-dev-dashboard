# `/api/scenes` HTML 503 diagnosis

## Finding

The observed HTML **503** originates upstream of the dashboard (Platform API,
its gateway, or a configured custom API host). The dashboard then incorrectly
turns that upstream 503 into its generic local 500 response. The failure is
therefore two separate facts:

1. **Availability/source failure:** the upstream server responded to the
   authenticated Scene-list request with HTTP 503 and an HTML body. A valid
   request reached an HTTP-serving upstream; this is not a browser route,
   session-authentication, MSW, or Next.js middleware failure.
2. **Dashboard error-mapping defect:** the legacy Scene handler does not retain
   a non-JSON Axios response status, so users see local `500 Unknown error from
   Vertex API` instead of a safe, actionable 503.

## Exact execution path

For `/api/scenes?pageSize=25`:

1. `src/pages/api/scenes.ts:get` obtains a session-built `VertexClient`.
2. It transforms the local request to the upstream URL
   `${client.config.basePath}/scenes?page[size]=25&fields[scene]=...` and calls
   `client.axiosInstance.get(...)` directly. The local `pageSize=25` is valid;
   it becomes standard JSON:API `page[size]=25`.
3. Axios rejects the upstream HTML 503. Such an error has
   `error.response.status === 503` and `error.response.data` as an HTML string.
   It is not the SDK's JSON:API `VertexError`/`Failure` shape.
4. The Scene catch block logs `error as VertexError` and evaluates only
   `error.vertexError?.res`. That field is absent for the raw Axios path, so it
   returns `ServerError` (`{ status: 500, message: "Unknown error from Vertex
   API." }`).

No HTML is sent to the browser; it is only logged by the server-side SDK error
logger. The status loss is the dashboard bug, while the original 503 remains an
external Platform/gateway availability (or custom-host routing) condition to
investigate.

## `makeCall`, `toErrorRes`, and framework behavior

- This Scene **GET does not call `makeCall`**. It has been a direct Axios path
  since scene partial-filter work.
- `makeCall` has the same general limitation: it returns
  `axiosError.response.data` cast as `Failure`. For an HTML body that is a
  string, `isErrorFailure` is false and callers cannot recover the upstream
  status.
- `toErrorRes` is intentionally a JSON:API failure converter: it requires an
  `errors` Set/array and reads `errors[0].status/title`. Passing HTML (or any
  non-JSON error body) yields its generic 500 fallback.
- The newer `createVertexRoute.toRouteError` similarly preserves structured
  JSON:API failures only; a raw Axios HTML 503 reaches its generic 500 fallback.
  That framework behavior does **not** currently govern the legacy
  `/api/scenes` handler.

## Change provenance / PR #326 assessment

PR #326 (`a2e3250`, test-harness merge) did not modify
`src/pages/api/scenes.ts`, `src/lib/vertex-api.ts`, `src/lib/api.ts`, or
`src/lib/api/route.ts`. It cannot directly create this upstream Scene request
or 503.

The relevant prior change is upstream commit `9ae445e` / PR #335 (2026-07-14):
it upgraded the API client 0.43.0 → 0.44.0 and changed Scene listing from
`client.scenes.getScenes(...)` to the current explicit Axios serialization so
typed partial-match filters render as canonical
`filter[field][contains]` parameters. The current manual request has the same
base path, `/scenes` route, OAuth bearer intent, page parameter, and sparse
field intent as the SDK call; with no filter, `pageSize=25` itself is not a
plausible malformed-request cause of an HTML 503.

PR #335 is worth comparing in a non-production environment only if the 503 is
reproducible and scoped to Scene listing: compare its explicit URL/headers with
the SDK-generated `getScenes` request while never logging bearer credentials.
An HTML 503 nevertheless indicates the response was generated by the remote
gateway/service rather than by filter parsing in this dashboard.

## Safe next steps

### External/operational investigation

1. Record the upstream host, response `Date`, request/correlation ID, and
   `Retry-After` header if present — **never** Authorization or HTML body — and
   check Platform/gateway health and logs for the same window.
2. Confirm the selected environment/custom `networkConfig.apiHost` is the
   intended Platform API host. A bad custom load-balancer route can also return
   an HTML 503.
3. Retry only after the advertised retry window (or a brief manual interval).
   Compare another read-only Platform endpoint using the same session to decide
   whether this is Scene-service-specific or a broader API-host outage.

### Narrow safe dashboard fix (separate implementation task)

Do **not** forward the HTML error body. Add a sanitizer that first preserves a
structured JSON:API `Failure` through `toErrorRes`, then maps an Axios
`response.status` in the 400–599 range to that same status with a fixed,
non-sensitive message. For example, the Scene GET can map 503 to:

```ts
{ status: 503, message: "Vertex Platform is temporarily unavailable. Try again shortly." }
```

Map other unstructured statuses to a fixed `Vertex API request failed.`
message; leave transport/no-response errors as the current local 500. A
shared helper may later be adopted by `toRouteError` and `makeCall`, but start
with a tested Scene-only change to avoid altering every legacy handler's public
error contract. Add Node MSW regression cases for both:

- JSON:API `{ errors: [...] }` 503 preserves the upstream title/status; and
- `Content-Type: text/html` 503 returns local status 503 with the fixed message,
  proving the HTML body never appears in JSON, logs asserted by tests, or the
  browser.
